You are working on the NightVex messenger project based on Tinode.

Main goal:
Turn this project into a stable self-hosted messenger with web, Android, and later iOS support. The app must support normal Telegram-like 1-on-1 messaging, group chats, easier user discovery, QR sharing, Android push notifications, and 1-on-1 voice/video calls.

Do not rewrite the whole project. Work incrementally. Make small, reviewable commits or change groups. Before changing behavior, inspect the existing implementation and document what already works, what is broken, and what needs to be added.

Project priorities:

1. Local development first
2. Fix broken or unsafe production issues
3. Make 1-on-1 messaging work properly
4. Fix group chat permissions
5. Fix user search and discovery
6. Fix QR code sharing and joining
7. Enable push notifications for Android
8. Enable 1-on-1 voice and video calls
9. Add geolocation sharing if available, otherwise implement safe MVP location sharing
10. Prepare Android app preview and build instructions
11. Keep production deployment separate from local development

Important safety rules:

* Do not commit real secrets.
* Do not commit .env files with real values.
* Do not commit Firebase service accounts, google-services.json, GoogleService-Info.plist, private keys, signing keys, database dumps, SSL certificates, or TURN credentials.
* Use .env.example or placeholder config files.
* Never log passwords, access tokens, private messages, phone numbers, emails, exact user locations, Firebase tokens, or auth credentials.
* Do not delete production data.
* Do not modify the production VPS directly unless explicitly instructed.
* Preserve existing Tinode protocol compatibility unless there is no other option.
* Prefer configuration changes over protocol changes.
* Preserve Russian localization and add Russian UI text where needed.

Phase 5 — Fix group chat permissions

Problem:
After joining a chat/group, users cannot write.

Investigate:

* Is the group actually a read-only channel?
* Are permissions/ACLs set incorrectly?
* Are new joiners missing write permissions?
* Are default access modes wrong?
* Are users joining as read-only subscribers instead of members?
* Is the UI creating channels instead of normal writable groups?

Fix:

* Normal groups must allow joined members to write by default.
* Channels may remain read-only if intentionally created as channels.
* Add a clear distinction in UI/config between “Group” and “Channel”.
* Ensure default group permissions include read + write for members.
* Do not give admin/owner permissions to normal users.
* Add tests or manual test steps.

Acceptance criteria:

* User creates a normal group.
* User invites or shares join link/QR.
* New user joins.
* New user can send messages.
* Existing users can see the message.

Phase 6 — QR code sharing and joining

Problem:
QR code in the app does not work for sharing/adding to chats.

Inspect:

* QR generation code
* QR scanning code
* Deep link format
* Invite link format
* Android intent filters
* Web route handling
* Permissions for camera access
* Whether QR contains a usable user ID, topic ID, invite token, or URL
* Whether QR points to production or local URLs incorrectly

Implement or fix:

* User profile QR: lets another user add/open direct chat.
* Group invite QR: lets another user join a group.
* Web fallback: QR link opens web app if mobile app is not installed.
* Android deep link: QR link opens NightVex app if installed.
* Safe token behavior: do not expose private auth tokens in QR.
* Expiring invite tokens if supported. If not supported, document limitation.

Recommended QR formats:

* nightvex://user/<public-user-id-or-username>
* nightvex://topic/<topic-id-or-invite-token>
* https://vshp-project.ru/invite/user/<public-id>
* https://vshp-project.ru/invite/group/<invite-token>

Acceptance criteria:

* User A shows profile QR.
* User B scans QR and can start/open 1-on-1 chat.
* Group admin/member shows group invite QR.
* User B scans QR and joins the group with write permission.
* Invalid/expired QR shows a friendly error.
